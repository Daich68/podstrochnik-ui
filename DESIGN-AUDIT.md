# Дизайн-аудит «проект подстрочник»

Аудит React-сайта (main page + post page + admin). Прочитаны все файлы `src/` (страницы, CSS, utils, App.tsx, index.css, public/index.html). Задача read-only — правки не вносились, ниже только находки и предложения.

**Общий вердикт.** Основа сильная: оливковая рама `#7a7a26`, один шрифт RG-StandardMedium, лаконичный хедер, цветные карточки — минимальная редакторская идентичность считывается. Основные проблемы — не «плохой дизайн», а недоделанные состояния (загрузка, пустой поиск, мобайл-hover), случайные элементы (emoji-иконки, транслит в заглушках, английский футер) и негарантированный контраст поверх произвольных цветов постов. И почти ничего в UI пока не говорит о самом «подстрочнике» как концепции.

## 1. Визуальный дизайн

### 1.1. Hover-only заголовки карточек — недоступны на тач-устройствах — **HIGH**
`Main.css:106–119`: `.post-info { opacity: 0 }` + `.post-card:hover .post-info { opacity: 1 }`.

**Почему важно.** На телефоне (а это, вероятно, основной трафик из Telegram) hover не существует: пользователь видит сетку разноцветных прямоугольников с картинками и ни одного названия/автора. Идентификация поста возможна только вслепую. Это одновременно и accessibility-проблема (информация недоступна без мыши), и удар по редакторской сути: имена авторов — контент, а не украшение.

**Фикс.** Не ломая десктопный hover, показать подпись на устройствах без hover:

```css
/* десктоп как есть */
@media (hover: none) {
    .post-info {
        opacity: 1;
        position: static;          /* подпись под картинкой, а не поверх */
        background: none;
        color: inherit;            /* цвет решается п.1.2 */
        padding: 8px 10px 10px;
    }
}
```
Вариант «подпись под картинкой» ещё и концептуально точнее — это буквально подстрочник (см. раздел 3). Можно применить его и на десктопе: строка появляется по hover *под* изображением (карточка чуть вырастает), вместо чёрной плашки поверх.

### 1.2. Контраст поверх произвольных цветов — **HIGH**
- Карточка: белый текст на `rgba(0,0,0,0.7)` — ок, но если убрать плашку (п.1.1), нужен расчёт.
- Страница поста (`PostPage.css`): фон — произвольный `post.color`, при этом кнопки топ-бара `color: black`, hover `#f0f0f0` (белый!), автор `#555`, дата `#777`, слик-точки — чёрные по умолчанию. На тёмном цвете поста (админ может выбрать любой, дефолт вообще `#000000` в `EditAddCore.tsx:23`) всё сливается; hover-состояние кнопок наоборот рассчитано на тёмный фон. Логика инвертирована и работает только на светлых цветах.

**Почему важно.** Один тёмный пост — и навигация невидима. WCAG-контраст сейчас — лотерея, которую крутит админ color-picker'ом.

**Фикс.** Одна утилита относительной яркости, применяемая на PostPage и в карточках:

```tsx
// utils/Style.tsx
export const isDarkColor = (hex: string): boolean => {
    const n = parseInt(hex.slice(1), 16);
    const r = (n >> 16) & 255, g = (n >> 8) & 255, b = n & 255;
    return (0.299 * r + 0.587 * g + 0.114 * b) < 140;
};
```
```tsx
// PostPage.tsx
const dark = isDarkColor(post.color);
<div className={`post-container ${dark ? "on-dark" : "on-light"}`} ...>
```
```css
.on-dark  { color: #f4f4ee; } .on-dark  .helpful-links-btn { color: #f4f4ee; }
.on-light { color: #1a1a1a; } .on-light .helpful-links-btn { color: #1a1a1a; }
.on-dark .post-author, .on-dark .post-time { color: rgba(255,255,255,.65); }
```
Плюс перекрасить слик-точки через тот же класс. Альтернатива-минимум: ограничить палитру в админке набором проверенных светлых цветов (валидация уже есть — `Yup.matches`, легко сузить).

### 1.3. Типографическая шкала: инверсия иерархии — **MEDIUM**
- Главная: `h1` = 3rem (заголовок сайта).
- Страница поста: `.post-title` = **1rem** (`PostPage.css:96`) — заголовок произведения на его собственной странице мельче, чем подпись в hover-карточке выглядит по весу. Автор — `#555` без размера.

**Почему важно.** Страница поста — «разворот книги», её главный текст — название и автор. Сейчас они выглядят как служебная подпись, и вся сцена держится только на картинке.

**Фикс.** Простая шкала из трёх ступеней, согласованная с главной:

```css
.post-title  { font-size: clamp(1.4rem, 3vw, 2.2rem); font-weight: normal; margin-bottom: .25rem; }
.post-author { font-size: 1rem; opacity: .7; margin-bottom: 1.5rem; }
```
(`font-weight: bold` у single-weight шрифта RG-StandardMedium даёт faux-bold — синтетическое ужирнение браузером, оно грязнит рисунок буквы; лучше размер/расстояние вместо жирности.)

### 1.4. Насколько оливковая рама держит произвольные цвета — **MEDIUM**
`#7a7a26` — тёплый, приглушённый, хорошо «гасит» пестроту карточек: сетка выглядит как коллекция закладок на сукне. Это работает. Два слабых места:
1. Цвета карточек ничем не нормированы — админ может выбрать неон или цвет, почти равный фону (карточка «растворится»: у карточки нет ни рамки, ни тени).
2. Переход главная → пост резкий: из оливкового в произвольный цвет всей страницей.

**Фикс.** (а) Тонкая внутренняя граница карточки, чтобы карточка не терялась при близком к фону цвете: `box-shadow: inset 0 0 0 1px rgba(0,0,0,.08);` — незаметно на контрастных, спасает на близких. (б) В админке — подсказка/пресет палитры (десяток «редакторских» цветов). (в) Переход фона уже анимирован (`transition: background-color .3s`) — можно инициализировать `.post-container` оливковым до загрузки поста, тогда переход станет мягким «перелистыванием».

### 1.5. Border-radius: три разных значения — **LOW**
Карточки и картинки — 10px, поиск — 8px на десктопе и **2px** на мобайле (`Main.css:37,70`), панель ссылок — 10px, fullscreen-картинка — 10px.

**Фикс.** Одна переменная:
```css
:root { --radius: 10px; }
```
и везде `border-radius: var(--radius)`. 2px на мобайле — явно случайность.

### 1.6. Футер — **MEDIUM**
`Main.tsx:104–108`, `Main.css:156–178`:
- Подписи на английском («design:», «founder and chief editor:») при полностью русском интерфейсе — стилистический разнобой.
- `tg@podstrochnik_project` — «tg» слиплось со ссылкой, нет пробела/двоеточия (`Main.tsx:107`).
- `.footer-divider` имеет `margin: 15px 0` на всю ширину, тогда как сетка имеет `padding: 20px` — линия шире контента, выглядит чужой.
- Футер по центру, тогда как хедер — flex слева; вертикальный ритм (15px / 10px / 20px) не совпадает с сеткой.

**Фикс.**
```css
.footer-divider { margin: 15px 20px; }  /* в створ сетки */
.site-footer    { text-align: left; padding: 10px 20px 24px; }
```
и микрокопия: «дизайн — katyamezentseva.com · главный редактор — alialiev.com · tg: @podstrochnik_project». Ещё лучше — оформить футер как сноску (см. 3.2).

### 1.7. Hover-scale карточек — **LOW**
`transform: scale(1.05)` на карточке в плотной masonry: соседние карточки перекрываются увеличенной, картинка слегка мылится, курсорная «дрожь» на границах колонок.

**Фикс.** Более редакторский жест — не рост, а лёгкий сдвиг/подчёркивание: `transform: translateY(-3px)` + тень, либо scale(1.02). И добавить `@media (prefers-reduced-motion: reduce) { .post-card { transition: none } }`.

### 1.8. Дублирующийся/мёртвый CSS — **LOW**
- `PostPage.css:70–72, 86–88` содержит правила `.masonry-grid` — на странице поста сетки нет, это осколки копипасты.
- `Main.css:18–21` и `:52–54`: `column-count: 2` задан и в базовом правиле, и в мобильном медиа-запросе — дубль.
- `PostPage.css:41–49`: два подряд одинаковых селектора `.helpful-links-btn:hover`.
- Вложенный CSS (`.search-icon img {}`) — фича CSS Nesting; в CRA-сборке без PostCSS-плагина работает только в свежих браузерах. Проверить таргеты или развернуть в плоские селекторы.

## 2. UX

### 2.1. Отсутствующие состояния — **HIGH**
- **Загрузка поста**: `PostPage.tsx:39` — `if (!post) return <p></p>;` — белая пустая страница (даже без оливкового фона, т.к. фон задан на `.post-container`). При медленном API это выглядит как поломка.
- **Загрузка главной**: до ответа API — оливковый экран с заголовком и ничего; при ошибке API — то же самое навсегда (ошибка уходит только в `console.error`, `Main.tsx:27`).
- **Пустой поиск**: запрос без результатов — просто исчезнувшая сетка, футер подпрыгивает под хедер.
- **`/about` и `/search`**: `MockPage` выводит «здесь скоро будет страница poiska» — транслит в пользовательском тексте (`App.tsx:28–29`). Ссылок на них из UI нет, но страницы доступны по URL и индексируются.

**Фикс (в духе минимализма, заодно работает на концепцию):**
```tsx
// главная
{filteredPosts.length === 0 && posts.length > 0 && (
    <p className="empty-note">* по запросу «{searchQuery}» ничего не нашлось</p>
)}
```
```tsx
// пост — держим оливковую раму до прихода данных
if (!post) return <div className="post-container" style={{ backgroundColor: "#7a7a26" }} />;
```
Ошибку сети — одной строкой в том же тоне: «* не получилось загрузить архив — попробуйте обновить». Заглушки `/about` переписать по-русски («страница «о проекте» готовится») или закрыть редиректом на главную.

### 2.2. Emoji-кнопки топ-бара поста — **HIGH**
`PostPage.tsx:57–63`: ↩️ «на главную», ↕️ «полный экран», ℹ️ «полезные ссылки».

**Почему важно.** (а) Эстетика: цветные глянцевые emoji — единственный «неавторский» графический элемент на странице; они спорят с серифом и заливкой цветом. (б) Кроссплатформенность: на Windows/Android/iOS эти глифы выглядят совершенно по-разному (и ↕️ на некоторых системах — чёрная стрелка без рамки). (в) Семантика: ↕️ не читается как «полный экран», ℹ️ означает «информация», а не «ссылки», и при этом та же ℹ️ используется для обратного действия «к карточкам». (г) На мобайле текст убирается и остаётся только emoji — кнопка ~16px без подписи и без `aria-label`.

**Фикс.** В проекте уже есть свой графический язык — рисованные SVG (`lupa.svg`, `arrow.svg`). Три монохромные SVG-иконки в той же линии (стрелка-возврат, рамка разворота, астериск/сноска для ссылок) закроют всё. Либо ещё минимальнее — только текст:
```tsx
<button className="helpful-links-btn" aria-label="на главную">
    <Link to="/">← на главную</Link>
</button>
<button ... aria-label="полный экран">развернуть</button>
<button ... aria-label="полезные ссылки">{showLinks ? "к карточкам" : "сноски *"}</button>
```
Типографские знаки (←, *, ×) вместо emoji — самый дешёвый и самый «редакторский» вариант. На мобайле не прятать подпись целиком, а сокращать («главная», «экран», «*») — иначе тап-таргеты 16px при норме 44px.

### 2.3. Карточка-ссылка со слайдером внутри — **MEDIUM-HIGH**
`Main.tsx:85–99`: весь `.post-card` — `<Link>`, внутри — slick с автоплеем и включённым swipe.

**Почему важно.** На тач-устройстве свайп по карусели легко распознаётся как тап → случайный переход на пост. Автоплей всех карточек разом (5с, `sliderSettingsV2Main`) делает сетку «мигающей», отвлекает от чтения и грузит CPU на длинной странице; к тому же autoplay продолжает крутиться и вне вьюпорта.

**Фикс.** Варианты по возрастанию затрат:
1. Выключить autoplay на главной вовсе — карусель листается пользователем, при hover можно запускать (`autoplay: false`, on mouseenter — `slickPlay()`), сетка сразу становится спокойной.
2. Оставить autoplay, но выключить swipe/draggable на главной (`swipe: false, draggable: false, touchMove: false`) — тогда любой жест по карточке однозначен: тап = переход.
3. Показать на карточке только первую (обложечную) картинку, а карусель оставить странице поста — концептуально чисто: обложка снаружи, разворот внутри.

### 2.4. Поиск — **MEDIUM**
`Main.tsx:39–81`, `Main.css:7–13, 28–47`:
- Иконка-лупа на мобайле — `height: 0.7rem` ≈ 11px: тап-таргет в 4 раза меньше минимума (44×44).
- При открытии инпут **не получает фокус** — на мобайле нужен второй тап, чтобы вызвать клавиатуру.
- Закрытие — только повторным кликом по стрелке; Escape и клик мимо не работают. Иконке-стрелке задан `alt="Search"`, хотя действие — «закрыть».
- Нет `:focus-visible`, рамка инпута закомментирована — на некоторых системах поле почти невидимо.
- Поиск идёт по `post.title`/`author_name`, которые приходят из Quill **с HTML-разметкой**: запрос «em» или «span» совпадёт с тегами, а слово, разорванное inline-тегом, — не найдётся. Разметку надо стрипать до сравнения.
- В `utils/Search.tsx` уже есть готовая функция поиска, но `Main.tsx` её не использует — дублирование логики.
- Роут `/search` — мёртвая заглушка, при том что реальный поиск живёт на `/?search=` (хорошо, что state в URL — расшариваемо; это плюс, сохранить).

**Фикс (ядро):**
```tsx
const inputRef = useRef<HTMLInputElement>(null);
const toggleSearch = () => {
    setIsSearchOpen(v => !v);
    if (!isSearchOpen) setTimeout(() => inputRef.current?.focus(), 300); // после анимации
    else { /* очистка как сейчас */ }
};
const strip = (html: string) => html.replace(/<[^>]+>/g, "").toLowerCase();
posts.filter(p => strip(p.title).includes(q) || strip(p.author_name).includes(q));
```
```css
.search-icon { padding: 12px; margin: -12px; }          /* растим тап-таргет без сдвига макета */
.search-input:focus-visible { outline: 2px solid rgba(255,255,255,.8); outline-offset: 2px; }
```
Плюс `onKeyDown={e => e.key === "Escape" && toggleSearch()}` на инпуте и `aria-label="закрыть поиск"` на стрелке.

### 2.5. Fullscreen-просмотрщик — **MEDIUM**
`PostPage.tsx:91–98`, `PostPage.css:124–177`:
- Кнопка «полный экран» всегда открывает `openFullscreen(0)` — первую картинку, даже если читатель дошёл до пятой в слайдере (клик по слайду закомментирован, `PostPage.tsx:81`). Разрыв контекста.
- Нет клавиатуры: Escape не закрывает, стрелки не листают. Нет свайпа на тач.
- Скролл подложки не блокируется (`overflow: hidden` на body).
- `.prev-btn/.next-btn { position: sticky }` — в flex-оверлее это работает случайно; стрелки прижаты к картинке, а не к краям экрана, зона клика мала.
- Звук перелистывания (`page.wav`, volume 1) — отличная концептуальная деталь («перелистывание страницы» — буквально книжный жест), но: не срабатывает respect для `prefers-reduced-motion`/mute, и не играет при закрытии/открытии.

**Фикс.**
```tsx
useEffect(() => {
    if (fullscreenIndex === null) return;
    document.body.style.overflow = "hidden";
    const onKey = (e: KeyboardEvent) => {
        if (e.key === "Escape") closeFullscreen();
        if (e.key === "ArrowRight") nextImage();
        if (e.key === "ArrowLeft") prevImage();
    };
    window.addEventListener("keydown", onKey);
    return () => { document.body.style.overflow = ""; window.removeEventListener("keydown", onKey); };
}, [fullscreenIndex]);
```
Открывать с текущего слайда: хранить `currentSlide` через `afterChange` в настройках слика и передавать его в `openFullscreen(currentSlide)`; заодно вернуть клик по слайду (`onClick={() => openFullscreen(index)}` уже написан и закомментирован). Стрелкам — `position: fixed; top: 50%; padding: 20px;`.

### 2.6. Перемешивание постов при каждом визите — **MEDIUM**
`Main.tsx:23`: `[...data].sort(() => Math.random() - 0.5)`.

**Почему важно.** (а) Читатель, вернувшийся «к той зелёной карточке слева сверху», её не найдёт — пространственная память обнуляется каждый визит; после возврата со страницы поста сетка тоже пересобирается заново — дезориентация. (б) Технически `sort` со случайным компаратором — некорректная тасовка (смещённая, зависит от реализации сортировки).

**Фикс.** Сохранить кураторскую идею «архив без иерархии», но стабилизировать в рамках сессии:
```tsx
const seed = Number(sessionStorage.getItem("shuffleSeed") ?? sessionStorage.setItem("shuffleSeed", String(Date.now())) ?? sessionStorage.getItem("shuffleSeed"));
// Fisher–Yates c seeded PRNG (mulberry32) вместо sort(random)
```
Порядок случайный для каждого нового посетителя, но неизменен внутри сессии (вкладка/возврат «назад»). Альтернатива — перемешивать раз в день (seed = дата).

### 2.7. Мобильный опыт, сводно — **MEDIUM**
- 2 колонки на мобайле при hover-невидимых подписях = «немая» сетка (см. 1.1).
- Тап-таргеты: лупа 11px, emoji-кнопки ~16px, слик-точки мелкие.
- `isMobileDeviceV3` по `window.innerWidth` + resize-листенер — работает, но убирает текст кнопок (см. 2.2).
- `.helper-post { height: 10vh }` — костыль-распорка под слайдером; вероятно, компенсирует слик-точки — лучше задать честный `padding-bottom` контейнеру.
- Шапка: `justify-content: space-around` у `.site-title` при открытом поиске даёт непредсказуемые зазоры на узких экранах; `white-space: nowrap` у h1 + инпут 100px — на 320px экране возможно выдавливание.

## 3. Редакторская концепция: как выразить «подстрочник»

Сейчас о концепции говорит только название и звук страницы. Ниже — приёмы, каждый из которых укладывается в существующий минимализм (ничего не добавлять «поверх», всё — переозначивание уже существующих элементов).

### 3.1. Подпись карточки = подстрочная строка — **рекомендую в первую очередь**
Вместо чёрной плашки-оверлея (п.1.1) — строка *под* изображением, как подстрочный перевод под строкой оригинала: картинка = «оригинал», подпись = «перевод».
```css
.post-info {
    position: static;
    opacity: 1;
    background: none;
    padding: 6px 10px 10px;
    border-top: 1px solid rgba(0,0,0,.15);   /* линия строки */
}
.post-info h3 { margin: 0; font-size: .95rem; font-weight: normal; }
.post-info p  { margin: 2px 0 0; font-size: .8rem; opacity: .65; font-style: italic; } /* автор — курсивом, как глосса */
```
Это разом решает мобильную проблему hover и делает каждую карточку микро-высказыванием формата «строка / подстрочник». Если жалко hover-магию на десктопе — подпись может «выезжать» из-под картинки по hover (height-transition), на тач — видна всегда.

### 3.2. Футер и служебные тексты = сноски
Футер уже стоит под чертой — остаётся оформить её как сноску, а не как `<hr>`:
- заменить линию на короткую черту сноски (как в книгах — линия на треть ширины, слева):
```css
.footer-divider { margin: 24px 20px 12px; width: 120px; background: rgba(255,255,255,.35); }
.site-footer { text-align: left; padding: 0 20px 24px; }
```
- каждая строка футера начинается с маркера сноски: `* дизайн — …`, `** редактор — …`, `*** tg — …`.
- тот же язык — для всех служебных сообщений: «* ничего не нашлось», «* загружается…». Появляется единый голос «примечаний на полях».

### 3.3. Нумерация карточек как сносок — деликатно
В углу каждой карточки — маленький надстрочный номер (`¹ ² ³ …` или `[1]`), стабильный (по порядку публикации, не по shuffle):
```tsx
<span className="footnote-mark">{post.index}</span>
```
```css
.footnote-mark { position: absolute; top: 8px; left: 10px; font-size: .7rem; vertical-align: super; opacity: .55; }
```
Сетка начинает читаться как страница, рассыпанная на примечания. Номер же даёт читателю «адрес» поста — частично компенсирует shuffle (п.2.6). Внедрять аккуратно: если цвета карточек пёстрые, номер должен наследовать логику контраста п.1.2.

### 3.4. Микрокопия
- Плейсхолдер поиска: вместо «поиск...» — «искать по строкам…» или «автор, строка…».
- Кнопка «полезные ссылки» → «сноски» / «примечания» (и это честнее описывает содержимое: редактор, дата, ссылки).
- Кнопка «к карточкам» → «к строкам» или «назад к развороту».
- Заголовок в шапке уже строчными — хорошо; можно усилить: «проект подстрочник» с тонким подчёркиванием у слова «подстрочник» (line under line — буквально): `border-bottom: 1px solid rgba(255,255,255,.4)` на span.

### 3.5. Страница поста = разворот с глоссой
Сейчас: заголовок → автор → слайдер. Малой кровью:
- автор под заголовком курсивом с отступом-переносом, как подстрочный перевод (уже почти так — усилить типографикой из п.1.3);
- дата и редактор в панели ссылок уже поданы мелко — оформить их с маркерами сносок;
- звук `page.wav` — оставить, это лучший концептуальный элемент сайта; добавить его и на переключение слайдов основного слайдера (не только fullscreen), с уважением к `prefers-reduced-motion`.

### 3.6. Чего НЕ делать
Не добавлять орнаменты, рамки «под старину», виньетки, вторые шрифты. Идентичность держится на «одна гарнитура + цветовые поля + типографские знаки (* × ← ¹)». Всё выше — только переозначивание существующих элементов этими знаками.

## 4. Мелкие технико-дизайнерские проблемы

### 4.1. `UpdateFavicon` в теле рендера — **MEDIUM**
`Main.tsx:59` и `PostPage.tsx:41` — вызов с DOM-мутацией прямо в render. В StrictMode исполняется дважды, ломает чистоту рендера, а в PostPage вызывается **на каждый ререндер** (каждое переключение слайда/панели).
```tsx
useEffect(() => { UpdateFavicon("#7a7a26"); }, []);
// PostPage:
useEffect(() => { if (post) UpdateFavicon(post.color); }, [post]);
```

### 4.2. Нет focus-стилей / клавиатурной навигации — **MEDIUM**
Ни одного `:focus-visible` в проекте. Карточки-ссылки фокусируются (это `<a>`), но hover-подпись по фокусу не раскрывается — клавиатурный пользователь тоже не видит названий:
```css
.post-card:focus-visible .post-info { opacity: 1; }
.post-card:focus-visible { outline: 2px solid rgba(255,255,255,.8); outline-offset: 2px; }
.helpful-links-btn:focus-visible, .close-btn:focus-visible { outline: 2px solid currentColor; }
```
Кнопка «на главную» — `<button><Link/></button>` (`PostPage.tsx:57`) — интерактивный элемент внутри интерактивного: двойной таб-стоп и невалидный HTML. Оставить только `<Link className="helpful-links-btn">`.

### 4.3. `lang="en"` и меты — **LOW**
`public/index.html:2` — `lang="en"` на русскоязычном сайте (скринридеры читают русский текст английским голосом, hyphenation не работает). Поменять на `lang="ru"`. Там же опечатка в description: «автокомменатриями» → «автокомментариями» (уходит в поисковую выдачу и в OG-превью Telegram).

### 4.4. Alt-тексты — **LOW**
`Main.tsx:91` — `alt={"Post {id} - Image {n}"}`: техномусор для скринридера. Лучше `alt={stripHtml(post.title)}` для первой картинки и `alt=""` для остальных (декоративные в контексте ссылки, у которой уже есть текст). `Main.tsx:79` — у стрелки закрытия поиска `alt="Search"` (см. 2.4).

### 4.5. `dangerouslySetInnerHTML` без санитизации — **MEDIUM (безопасность)**
Заголовки, авторы, helpful_links рендерятся сырым HTML из админки (`InnerDangerous.tsx`, `Main.tsx:96–97`). Пока авторы контента доверенные — риск ограничен, но любая компрометация админки = stored XSS для всех читателей. `RemoveLinksFromHTML` убирает только `<a>`. Дешёвый фикс — DOMPurify на входе `InnerDangerous`.

### 4.6. Тап-таргеты (сводно) — **MEDIUM**
- лупа на мобайле: 11px (норма 44) — `Main.css:31–33`;
- emoji-кнопки на мобайле: ~16px без подписи;
- слик-точки поста: дефолтные 20×20 с 5px зоной;
- `×`/`‹`/`›` в fullscreen: `padding: 10px` при `font-size: 2rem` — терпимо, но у краёв экрана лучше 44px+.
Общий приём: `padding` растим, компенсируем отрицательным `margin`.

### 4.7. Разное — **LOW**
- `App.css`: глобальный `a { color: black }` — чёрные ссылки на оливковом фоне вне футера (если появятся) будут с контрастом 4.2:1 на грани; учитывать при добавлении контента.
- `index.css`: `@font-face` без `font-display: swap` — при медленной загрузке TTF текст невидим (FOIT); и стоит подложить `format('truetype')`-фолбэк на woff2 (TTF ~в 2 раза тяжелее).
- `iframe { display: none }` в `index.css` — глобально прячет ВСЕ iframe (видимо, костыль против CRA-оверлея); если в helpful_links когда-нибудь встроят YouTube — он молча не покажется.
- `GetPostByID` фильтрацией по дате не защищён (в отличие от `GetPost`) — прямой URL показывает неопубликованный пост. Возможно, задумано как «превью по ссылке», но стоит решить осознанно.
- `Main.tsx:102–109`: лишний фрагмент `<>…</>` вокруг футера; footer выводится даже при пустом результате поиска сразу под шапкой — с empty-state из п.2.1 это станет опрятно.
- `reportWebVitals()` вызывается вхолостую.
